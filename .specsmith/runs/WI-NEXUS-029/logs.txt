# WI-NEXUS-029 / REQ-109 — l1-nexus smoke overlay evidence

The new `docker-compose.smoke.yml` overlay pins the served model to
`Qwen/Qwen2.5-Coder-7B-Instruct-GPTQ-Int4` (~5 GB VRAM at batch=1) so
`scripts/nexus_smoke.py` with `NEXUS_LIVE=1` can produce real `ok: true`
evidence on a 12 GB developer GPU.

## Reproducible procedure

```
docker compose -f docker-compose.yml -f docker-compose.smoke.yml up -d
# wait for the model to load (60-120s on first pull, ~5s on warm cache)
$env:NEXUS_LIVE = '1'
py scripts/nexus_smoke.py | Tee-Object -FilePath .specsmith/runs/WI-NEXUS-029/live-ok.txt
docker compose -f docker-compose.yml -f docker-compose.smoke.yml down
```

When the container is running, `scripts/nexus_smoke.py` returns
`{"ok": true, "content": "ready", "latency_ms": <small>, "error": ""}`
which satisfies REQ-109. The file `live-ok.txt` (alongside this one) is
the captured payload from a real run on the dev workstation.

## Why an overlay rather than the production compose file

The production `docker-compose.yml` pins `Qwen2.5-Coder-32B-Instruct-GPTQ-Int8`
which needs ~20 GB VRAM. The dev workstation has 12 GB; the overlay swaps
to the 7B Q4 quant so we have *real* end-to-end evidence rather than only
documenting an honest skip. CI deployments with 24+ GB GPUs should use the
default compose file (no overlay) for the production model.
