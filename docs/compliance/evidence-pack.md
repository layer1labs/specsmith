# Compliance Evidence Pack

This guide explains what SpecSmith can generate for audit evidence and what the user organization must still provide.

## What SpecSmith can generate

- Requirements trace (requirement IDs, status, coverage)
- Test trace (test cases mapped to requirements)
- Work item history (state transitions, timestamps, ownership metadata)
- Agent action records (governance decisions and execution traces)
- Preflight decisions (accepted/needs clarification records)
- Verification results (equilibrium checks and validation outcomes)
- Audit ledger and trace chain artifacts
- Human review checkpoints recorded in process artifacts

## What users must still provide

- Legal/regulatory classification decisions
- Formal risk assessment and control mapping
- Human oversight policy and approval records
- Independent validation evidence for safety/performance claims
- Deployment controls (IAM, network, secrets, runtime hardening)
- Incident response and post-incident governance process

## Sample evidence export structure

```text
compliance-export/
  manifest.json
  requirements/
    requirements-trace.json
    requirements-summary.md
  tests/
    test-trace.json
    coverage-summary.md
  work-items/
    workitems.json
    lifecycle-summary.md
  governance/
    preflight-log.jsonl
    verify-results.json
    audit-report.json
  ledger/
    trace.jsonl
    ledger.jsonl
    trace-verify-report.json
  reviews/
    checkpoints.md
```

## Schema versioning note

Compliance export formats should include an explicit schema version field in exported manifests and key JSON payloads. Automated schema-version-field checks are tracked in #218.

## Disclaimer

Evidence generated by SpecSmith is supporting documentation, not a legal determination of compliance. Final accountability for regulatory conformance remains with the deploying organization and its qualified reviewers.
